Offline Mode

Providing a clear sense of context and interactive possibility when server connection is lost. In some cases of offline mode you may choose to allow data to be edited and pushed to the server once a connection is restored. These patterns provide some options for communicating the state to your users.

What It Achieves

Offline mode generally consists of ensuring application caching and offline storage. Providing this may allow your user to continue an experience with an application’s core logic and UI as well as capture user generated data when not connected to servers. If limitations result from offline usage, it may be necessary to provide contextual notifications to users.

How To Use

We’ve provided a code example for you to use and modify. We are giving you the full code needed to make this work on our public GitHub repository.

Example Code

Emphasize Non-Error Delay State

In the example of email, allowing users to compose content and have it save automatically helps make the experience feel functional even while offline. A simple notification can be used to indicate the plan of action once connection is established.

Allow and Distinguish Queued Content

In the example of a social application, a distinction between queued content and published content can be helpful. A status bar, grayed out content, or iconography acting as indication that queued content is waiting for a connection can further inform the user that their content exists but is not yet publicly published.

Timestamps for Highly Organic Data

In the example of a weather application, alerting the user while also providing a timestamp of when the application last updated can be helpful in indicating both a lack of connection and which information can still be useful. Also allowing access to previously loaded data, though not current, makes the app functional even while offline.
